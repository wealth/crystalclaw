## Project Overview

CrystalClaw is a multi-channel AI agent gateway written in Crystal. It connects messaging platforms (Telegram, Discord, etc.) to LLM providers through a central agent loop with tool use support.

## Build & Run Commands

```bash
# Build release binary
shards build --release --no-debug

# Initialize config and workspace (first time)
./bin/crystalclaw onboard

# Run in gateway mode (multi-channel listener)
./bin/crystalclaw gateway

# Send a direct message to the agent
./bin/crystalclaw agent -m "your message"

# Interactive agent mode
./bin/crystalclaw agent

# Debug mode (verbose LLM/tool output)
./bin/crystalclaw gateway -d
./bin/crystalclaw agent -d

# Manage cron jobs
./bin/crystalclaw cron list

# Docker
docker-compose up -d
```

There is no test suite — the project uses manual/integration testing.

## Architecture

### Request Flow

```
Channel (Telegram/Discord/etc.)
  → Message Bus (async channels)
    → Agent Loop
      → Session (conversation history)
      → LLM Provider (OpenRouter/Anthropic/OpenAI/Ollama/etc.)
      → Tool Execution (filesystem, shell, web, message)
      → Repeat until done
    → Message Bus (outbound)
  → Channel (send response + media)
```

### Key Subsystems

**`src/agent/loop.cr`** — Core orchestration. Builds LLM prompts with tool definitions, executes tool calls in a loop until the model returns a final response, handles failover between providers.

**`src/bus/bus.cr`** — Fiber-based async message bus. Channels are decoupled: each messaging platform produces `InboundMessage` and consumes `OutboundMessage` via the bus.

**`src/channels/`** — One file per platform. Each channel runs in its own fiber, polls or listens for messages, translates to/from the bus format. Telegram is the most complete.

**`src/providers/`** — LLM provider abstractions. All providers share a common interface. The agent uses failover: if a provider fails, it logs the reason and tries the next configured one.

**`src/tools/`** — Tools available to the agent: filesystem reads/writes (workspace-sandboxed), shell execution, web search (DuckDuckGo/Brave), code editing, and `MessageTool` for sending rich responses back to channels including media.

**`src/memory/`** — Two backends: `FileStore` (flat files in workspace) and `PostgresStore` (dot-notation key-value in PG). PG store seeds from workspace files on first use.

**`src/session/manager.cr`** — Per-user conversation history keyed by `channel:user_id`. Holds message history passed to the LLM on each turn.

**`src/config/config.cr`** — Central config struct, JSON-serializable, stored at `~/.crystalclaw/config.json` or in PostgreSQL. Dot-notation key paths mirror the struct hierarchy (e.g., `agents.defaults.model`).

**`src/skills/loader.cr`** — Loads skill definitions from the workspace directory at startup.

**`src/cron/`** — Scheduled job service. Jobs run in fibers; cron expressions and interval syntax both supported.

### Workspace

`~/.crystalclaw/workspace/` (or Docker-mounted `/workspace`) holds agent identity files (`AGENT.md`, `IDENTITY.md`, `SOUL.md`, `USER.md`) and a `memory/` directory. These are loaded into agent context. The `workspace/` directory in the repo contains the templates.

### Two Operating Modes

- **CLI mode** (`agent` command): Single-turn or interactive, no channels running.
- **Gateway mode** (`gateway` command): Starts all configured channels, the cron service, and the heartbeat service concurrently as fibers.

## Configuration

Config file: `~/.crystalclaw/config.json` (generated by `onboard`).

Key config sections:
- `agents.defaults` — model, workspace path, token limits, max tool iterations
- `channels` — per-platform tokens/settings (Telegram bot token, Discord webhook, etc.)
- `providers` — API keys for each LLM provider
- `tools` — web search backend and API key

When PostgreSQL is configured (`CRYSTALCLAW_POSTGRES_URL` env var or `memory.postgres_url` config key), config and memory are stored in PG instead of files, enabling multi-instance deployments.

## Deployment

**Docker:** `docker-compose up -d` — mounts workspace volume, optionally includes PostgreSQL.

**systemd:** `sudo ./install-service.sh` — builds binary, creates `crystalclaw` user, installs unit file.

The Docker runtime is `ubuntu:24.04` (not Alpine) to avoid musl libc compatibility issues with libpq.
